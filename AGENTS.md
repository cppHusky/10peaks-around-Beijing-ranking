# AGENTS.md

这是一个 Cloudflare Worker，用于统计“徒步强国京畿十峰挑战赛”参与队员的打卡数，并绘制相关排行榜。

你需要为管理员提供以下功能：

- 数据源的上传与存储；
- 数据处理和数据库更新；
- 绘制排行榜、查看并提供下载。

普通访问者访问本站只能浏览和下载排行榜，无权查得任何其它信息。

要求如下：

## 数据源的上传与存储

排名系统的数据收集自各领队出队过程中的活动参与表和十峰挑战赛的报名时间表，全部为 xlsx 表格的形式。表格由管理员上传，存储在 Cloudflare R2 当中。

### 活动参与表

活动参与表包含以下有用信息：

- G3：活动开始日期；
- G4：活动结束日期；
- D 列：电话，自 D7 起依次向下；
- H, I, J, ... 列：标题位于第 6 行，标题文本可能是 `打卡${somewhere}`，也可能直接是山峰名；只要标题文本包含[山峰列表](#山峰列表)中的某个山峰名，即视为该列对应此山峰。接下来每个单元格依次对应于 D 列电话号，如果单元格填充为绿色（可能存在不同的“绿色”，可以适当放宽颜色判据），表示打卡成功，否则未打卡。

### 十峰挑战赛报名时间表

十峰挑战赛报名时间表包含以下有用信息：

- A 列：序号，自 A7 起依次向下；
- C 列：用户昵称，自 C7 起依次向下；
- D 列：电话，自 D7 起依次向下；
- F 列：报名时间，自 F7 起依次向下。

其中序号是逻辑上的主键，不可重复；电话也不可重复，并在与活动参与表联立时作为核对字段使用。如存在重复，会在上传时中断，并提示重复者信息。解析时应忽略关键列为空的预留空行。

### 管理员界面

管理员界面将陈列现有数据表的名字，并支持管理员上传新文件、重新上传已有文件、下载已有文件和删除已有文件。上传过程应展示进度反馈；浏览器上传阶段显示真实传输进度，服务端解析和数据库更新阶段显示处理中状态。活动参与表支持批量上传，同名文件按重新上传处理。[十峰挑战赛报名时间表](#十峰挑战赛报名时间表)置顶。

管理员界面使用 token 验证身份，token 由开发者自行生成并写入环境变量。登录状态保存在 cookie 中。

## 数据处理和数据库更新

排名系统的数据源是 xlsx 表格，这些数据不便于直接使用，必须整理成数据库信息的格式，存储在 Cloudflare D1 当中。

这个数据库包含以下四张表：

- 十峰挑战赛报名表 `attendance`；
- 数据源总表 `raw`；
- 最终数据表 `final`；
- 数据源发布表 `sources`。

数据源文件与数据库更新需要保持应用层原子发布语义：上传时先将文件写入 R2 唯一对象 key，解析成功后再在同一次 D1 原子批处理中更新业务表和 `sources` 的 `ready` 记录。管理员列表、下载和删除只以 `sources.status = 'ready'` 的记录为准，不直接以 R2 列表作为业务真相；若中途失败，临时或孤儿 R2 对象不可见，不应影响排行榜。

### 十峰挑战赛报名表

`attendance` 表完全来自十峰挑战赛报名时间表，它包含以下字段：

- `serial` 表示报名序号，它是主键；
- `phone` 表示报名者电话，它不可重复，并用于与 `raw` 表按电话关联；
- `name` 表示报名者用户昵称，它可能包含中、西文，特殊字符和 emoji 等；
- `attend_time` 表示报名时间，它是一个日期时间，保留时分秒信息，用于排序；在与活动日期进行资格判断时，统一使用 `date(attend_time)` 参与比较。

每当数据源的十峰挑战赛报名时间表发生变更时，这个表都需要更新。

### 数据源总表

`raw` 表数据来自所有活动参与表的总和，并用一个字段表示本纪录来自哪份 xlsx。它包含以下字段：

- `source` 表示来源文件名。
- `start_date` 表示活动的起始日期，由文件的 G3 得来，它是一个日期。
- `end_date` 表示活动的结束日期，由文件的 G4 得来，它是一个日期。
- `phone` 表示报名电话，但它不能是 `attendance` 的外键，因为不是所有活动参与者都报名了十峰挑战赛。
- `mask` 表示 30 个山峰的打卡状态位。山峰与 bit 位的映射顺序固定为[山峰列表](#山峰列表)中的顺序；默认值为 `0`，若 H6, I6, ... 的标题文本包含某山峰名，且对应单元格填充为绿色，则将该山峰对应的 bit 置为 `1`。

数据源总表随数据源中的活动参与表更新，每当活动参与表有新上传、更新、删除时，都要把 `raw` 中有关数据进行相应的更新，以确保 `raw` 的数据总是与活动参与表同步。

### 最终数据表

`final` 表是对 `attendance` 和 `raw` 的汇总缓存。它描述的是，每个参与者是否在报名十峰挑战赛之后完成了某山峰的打卡；可以补报名十峰挑战赛，但补报名日期不能晚于活动结束日期的第二天，且补报名之前参加的打卡不能计入统计。由于活动参与表只提供活动起止日期而不提供精确打卡时间，系统在资格判断时统一使用 `date(attend_time)` 与活动日期比较，并以活动结束日期 `end_date` 的第二天作为该次活动是否计入统计的判据。`final` 仅存聚合结果，不重复存储序号和用户昵称；排行榜需要这些信息时，再与 `attendance` 表联结读取。它包含以下字段：

- `phone` 表示报名电话，它一定来自 `attendance`。
- `mask` 表示 30 个山峰的最终打卡状态位，山峰与 bit 位的映射顺序固定为[山峰列表](#山峰列表)中的顺序，默认值为 `0`。如果某一 bit 取 `1`，需要满足：`raw` 中此人有至少一条对应山峰的打卡记录，且 `date(attend_time)` 不晚于该记录的 `end_date` 的第二天。
- `total_count` 表示 `mask` 中取 `1` 的总数，用于排行榜排序和查询优化。

因此，一旦 `attendance` 和 `raw` 发生变化，`final` 中受影响参与者的聚合结果也需要重新核算。

### 数据源发布表

`sources` 表记录已经完成发布的数据源文件，包含以下字段：

- `kind` 表示数据源类型，取值为 `attendance` 或 `activity`。
- `name` 表示业务文件名；活动参与表使用原文件名，报名时间表使用固定名 `attendance.xlsx`。
- `r2_key` 表示实际 R2 对象 key。
- `size` 表示文件大小。
- `uploaded_at` 表示发布时间。
- `status` 表示发布状态；普通管理操作只展示和使用 `ready` 状态的数据源。

## 绘制排行榜

本项目已经实现过一个基础的绘制功能了，不过当时使用的山峰列表（和支线情况）不同于现在的版本，所以需要重新进行绘制。

此外，原本的最终数据表是以 xlsx 文件实现的，但现在我们有了 Cloudflare D1 数据库，需要更改一下数据加载的逻辑。

绘制排行榜应使用 `final` 与 `attendance` 联结后的 `serial`、`name`、打卡状态值和统计值，`phone` 不体现在表内。

排行榜展示汇总图和逐人打卡详表，不展示顶部数据块或领先者名单。打卡数量为 `0` 的参与者不显示。图中不展示序号，标题说明展示数据更新时间。详表展示排名、名字和完成打卡点名称；未完成东灵山者排名显示 `*`，且不计入排名序列。公开界面不提供 CSV 下载，应提供 PNG 下载，下载文件名带下载时间后缀。

汇总图保留连续色条表达，支线色条右端使用圆形端帽，并以轻微重叠形成圆角连接；若参与者完成东灵山，在进度条第一个格放置红色五角星；若完成任意一条支线的全部山峰，在对应支线的最后一格放置黑色对勾。若参与者完成全部山峰，则使用一条左端直角、右端圆角的贯通渐变进度条，渐变依次经过各支线颜色，并只保留开头红色五角星和结尾一个黑色对勾。

排序的规则是：打卡数量降序，序号升序。
